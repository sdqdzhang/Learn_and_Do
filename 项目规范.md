# Tiny-Devin 基础阶段开发规范 (V1.3)

本文档定义了 Tiny-Devin 的核心接口。

- **V1.1** 强化了类型安全、异常处理、容器生命周期管理。
- **V1.2** 引入"双态架构"：同一底座兼容**代码开发**与**哲学研究**两种 Agent 形态。
- **V1.3** 完整化为 **4 层基础架构**：**认知层 / 执行层 / 记忆层 / 编排层**，新增 `tools/`、`memory/`、`audit/` 三个模块，明确状态机语义与 Jinja2 prompt 引擎。

---

## 0. 设计理念

### 0.1 双态架构

| 模式 | TaskMode | Docker 用途 | Reviewer 关注点 |
|---|---|---|---|
| 代码开发 | `DEVELOPMENT` | 跑测试 / 修 Bug | 语法、单测 |
| 哲学研究 | `PHILOSOPHY` | 跑爬虫 / 数据建模 | 逻辑谬误、数据证伪 |

### 0.2 四层架构

```
┌───────────────────────────────────────────────────────┐
│  4. Orchestration Layer  (core/workflow.py)           │
│     状态机 + Planner + Reflector                       │
│     ▲ 调用以下三层                                     │
├─────────────────┬─────────────────┬───────────────────┤
│ 1. Cognitive    │ 2. Execution    │ 3. Memory         │
│  prompt_temp    │  runtime/exec   │  memory/session   │
│  utils/parser   │  tools/*        │  memory/vector    │
│  utils/llm_cli  │                 │                   │
└─────────────────┴─────────────────┴───────────────────┘
                            ▲
              core/schema, core/exceptions, core/audit
              （横切：协议 / 错误 / 审计）
```

**核心解耦原则**：
- **底层（schema / exceptions / llm_client）模式无关**。
- **中层（parser / executor / memory / tools）** 接收 `TaskMode` 字段但行为不分支，只让"插件"自己解释模式。
- **顶层（workflow / prompt_templates）** 是唯一根据模式切换"灵魂"的地方。

## 2. 协议层 (core/schema.py)

```python
# 核心枚举
class MessageRole(str, Enum):  SYSTEM, USER, ASSISTANT, TOOL
class FileAction(str, Enum):   WRITE, DELETE, READ
class TaskMode(str, Enum):     DEVELOPMENT, PHILOSOPHY
class EvidenceType(str, Enum): CODE_RESULT, WEB_SOURCE, DATA_FILE
class ToolStatus(str, Enum):   PENDING, SUCCESS, FAILED
class AgentState(str, Enum):   IDLE, THINKING, ACTING, OBSERVING, REFLECTING, DONE, FAILED

# 数据包
class ChatMessage:          role, content, thought?, tool_call_id?
class FileOperation:        file_path, content?, action
class Evidence:             type, source, summary, payload?
class ExecutionResult:      is_success, stdout, stderr, active_files, metrics, metadata, artifacts
class ToolCall:             id, name, args (dict)
class ToolResult:           call_id, status, output (Any), error?
class Plan:                 goal, steps: List[str], notes?
class Reflection:           plan_ref, observations, conflicts: List[Evidence], next_action: Literal["retry","revise","done"]
class TraceEvent:           ts, session_id, turn, state, kind, payload (dict)
```

`ChatMessage.thought` 用于 PHILOSOPHY 模式必填；`role=TOOL` 时 `tool_call_id` 必填。

## 3. 异常体系 (core/exceptions.py)

```
TinyDevinError
├── RetryableError
│   ├── CodeFormatError       解析失败
│   ├── MissingPathError      代码块缺路径
│   ├── LLMTimeoutError       LLM 重试预算耗尽
│   ├── EvidenceConflict      执行结果与论点冲突
│   ├── ToolError             工具调用失败但可重试
│   └── MemoryError           记忆层瞬态故障
└── FatalError
    ├── SandboxViolation      尝试越权指令
    ├── ContainerImageError   基础镜像缺失
    ├── ResourceExhausted     容器超限
    └── ConfigurationError    配置缺失或非法
```

## 4. 认知层 (Cognitive Layer)

### 4.1 prompt_templates.py — Jinja2 角色矩阵

```python
class SystemPromptBuilder:
    def __init__(self, role: AgentRole, mode: TaskMode,
                 tools: List[ToolSpec] = None, extra_ctx: Dict = None): ...
    def render(self) -> str: ...        # Jinja2 渲染
```

- 角色：`CODER` / `REVIEWER` / `PHILOSOPHER` / `INVESTIGATOR` / `PLANNER` / `REFLECTOR`
- 模板存放：模块内字典，按 `(role, mode)` 查表
- 自动注入：可用工具描述、输出格式规则、TaskMode 特定约束（如 PHILOSOPHY 必出 `<thought>`）

### 4.2 utils/parser.py — 多块严格解析

支持四种 block，可共存：

1. `<file path="...">...</file>` — 代码 / 文档 / 数据
2. `<thought>...</thought>` — 思辨（PHILOSOPHY 必选）
3. `<tool name="..."><![CDATA[ {json args} ]]></tool>` 或 `<tool name="...">{json}</tool>` — 工具调用
4. ```` ```json ... ``` ```` — 结构化输出（Plan / Reflection）

返回：`ParsedOutput(files, thoughts, tool_calls, json_blocks)`。

### 4.3 utils/llm_client.py — 已完成 (V1.2)

无改动。

## 5. 执行层 (Execution Layer)

### 5.1 runtime/executor.py

接口：

```python
class Executor:
    def __init__(self, config: ExecutorConfig = None): ...
    def run(self,
            files: List[FileOperation],
            command: List[str],
            *,
            extra_pip: List[str] = None,
            timeout: int = 60) -> ExecutionResult: ...
    def close(self): ...
```

行为：
1. `FileOperation` 落到 `WORKSPACE_DIR`（宿主机）。
2. Volume 挂载 `WORKSPACE_DIR` → 容器 `/workspace`，命令在 `/workspace` 下运行。
3. 应用 `CONTAINER_MEMORY_LIMIT` / `CONTAINER_CPU_LIMIT` / `timeout`。
4. 运行后 `find . -maxdepth 2 -not -path '*/.*'` 取文件清单，diff 后填充 `active_files` 与 `artifacts`。
5. 失败按规范 §3 抛对应 Fatal 异常。

### 5.2 core/session_manager.py

```python
class SessionManager:
    def __init__(self, config: SessionConfig = None): ...
    def start(self) -> str:              # 返回 session_id
    def heartbeat(self) -> None: ...
    def stop(self) -> None: ...
    @property
    def executor(self) -> Executor: ...
```

行为：
- 单容器单会话，session_id 是 `{prefix}-{uuid8}`。
- 后台线程每 `HEARTBEAT_INTERVAL` 检测空闲；超 `IDLE_TIMEOUT` 强停。
- 与 `Executor` 1:1 绑定。

### 5.3 tools/ — 统一工具接口

```python
# tools/base.py
class ToolSpec(BaseModel):  name, description, args_schema (JSON Schema)
class Tool(Protocol):
    spec: ToolSpec
    def call(self, args: dict) -> ToolResult: ...

# tools/registry.py
class ToolRegistry:
    def register(self, tool: Tool) -> None: ...
    def get(self, name: str) -> Tool: ...
    def list_specs(self) -> List[ToolSpec]: ...
    def invoke(self, call: ToolCall) -> ToolResult: ...
```

具体工具：

| 工具 | 文件 | 输入 | 输出 |
|---|---|---|---|
| `file_read` / `file_write` / `file_list` | `tools/file_io.py` | path, content? | 文件内容 / 路径列表 |
| `web_search` | `tools/search.py` | query, k | 搜索结果列表 |
| `python_repl` | `tools/repl.py` | code, timeout? | stdout / stderr |
| `rag_query` | `tools/rag.py` | query, k | 文档片段列表 |

约束：所有工具的 `args` 必须经过 `args_schema` 校验；失败抛 `ToolError`（可重试）或 `SandboxViolation`（熔断）。

## 6. 记忆层 (Memory Layer)

### 6.1 memory/session_context.py — 短期记忆

```python
class SessionContext:
    def __init__(self, max_tokens: int = 8000,
                 summarizer: Callable[[List[ChatMessage]], str] = None): ...
    def add(self, message: ChatMessage) -> None: ...
    def messages(self) -> List[ChatMessage]: ...
    def compress_if_needed(self) -> None: ...   # 触发摘要
    def token_count(self) -> int: ...
```

行为：
- Token 计数优先用 `tiktoken`，找不到合适 encoder 时退化为字符数 / 4。
- 超过 `max_tokens` 时调用 `summarizer` 把旧消息浓缩为单条 `SYSTEM` 消息。
- `summarizer` 默认 `None` —— 由 `workflow` 注入一个用 `LLMClient` 实现的版本。

### 6.2 memory/vector_store.py — 长期记忆

```python
class VectorStore:
    def __init__(self, persist_dir: str, collection: str): ...
    def upsert(self, texts: List[str], metadatas: List[dict] = None,
               ids: List[str] = None) -> List[str]: ...
    def query(self, text: str, k: int = 5) -> List[VectorHit]: ...
    def delete(self, ids: List[str]) -> None: ...
```

- 底层 ChromaDB，模块内**懒加载**（首次 upsert/query 时才 import），避免冷启动慢。
- 嵌入函数走 ChromaDB 默认（`all-MiniLM-L6-v2`），可通过环境变量覆盖。
- 模式无关：DEVELOPMENT 存代码片段 / 单测错误；PHILOSOPHY 存论据 / 反论据。

### 6.3 runtime/workspace/ — 文件级记忆

继续作为 Shared Directory，与容器 volume 双向同步。`executor.py` 是唯一写入者，`tools/file_io.py` 是 Agent 操纵它的入口。

## 7. 编排层 (Orchestration Layer)

### 7.1 core/workflow.py — 状态机

```
       ┌───────┐
       │ IDLE  │
       └───┬───┘
           │ run()
           ▼
       ┌────────┐  generate plan / next step
       │THINKING│◀──────────┐
       └───┬────┘           │
           │ has tool calls │
           ▼                │
       ┌────────┐           │
       │ ACTING │ run tools │
       └───┬────┘           │
           ▼                │
       ┌─────────┐          │
       │OBSERVING│ collect  │
       └───┬─────┘          │
           ▼                │
       ┌──────────┐         │
       │REFLECTING│─────────┘ continue?
       └───┬──────┘
           │ done / max_turns / fatal
           ▼
     ┌──────┴──────┐
     ▼             ▼
  ┌─────┐      ┌──────┐
  │ DONE│      │FAILED│
  └─────┘      └──────┘
```

```python
class Workflow:
    def __init__(self,
                 llm: LLMClient,
                 executor: Executor,
                 tools: ToolRegistry,
                 context: SessionContext,
                 trace: TraceLogger,
                 mode: TaskMode,
                 max_turns: int = 10): ...
    def run(self, user_prompt: str) -> WorkflowResult: ...
```

行为：
- 状态转移由内部驱动，外部只调 `run()`。
- 每次状态切换写一条 `TraceEvent`。
- `THINKING` 调 LLMClient；`ACTING` 调 ToolRegistry；`OBSERVING` 收集 `ToolResult`；`REFLECTING` 用 LLM 产出 `Reflection`，决定 retry/revise/done。
- 捕获 `RetryableError`：写入 trace、构造反馈消息、回到 `THINKING`。
- 捕获 `FatalError`：写入 trace、状态切到 `FAILED`、抛给调用方。

### 7.2 core/audit.py — 轨迹日志

```python
class TraceLogger:
    def __init__(self, log_dir: str, session_id: str): ...
    def log(self, kind: str, payload: dict, *, state: AgentState, turn: int) -> None: ...
    def close(self) -> None: ...
```

- 每个 session 一个 JSON Lines 文件：`runtime/traces/{session_id}.jsonl`。
- 字段：`ts, session_id, turn, state, kind, payload`。
- `kind` 枚举：`prompt`, `response`, `tool_call`, `tool_result`, `exec_result`, `state_change`, `error`, `reflection`。

## 8. 执行器生命周期策略 (沿用 V1.1)

- **镜像**：`tiny-devin-base:latest`（预装 requests/numpy/pandas/matplotlib/scipy/pytest）+ 运行时 `pip install`。
- **清理**：`SessionManager` 心跳每 10 分钟，空闲 1 小时强停容器。
- **隔离粒度**：单容器单会话（MVP）。
